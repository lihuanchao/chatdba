
### 步骤 5：综合分析

综合执行计划与元数据，提供具体、可操作的优化建议：

#### 1. 执行计划深度诊断 (Execution Plan Analysis)

- 执行计划为测试环境空表下获取的，请逐行分析执行计划中的每个节点，重点关注以下字段，无需关注rows、filtered：
  - `id`: 表示查询中各个操作的顺序和层次。id 值越大，优先执行。
  - `select_type`: 显示查询类型，包括 SIMPLE（基本查询）、PRIMARY（主查询）、SUBQUERY（子查询）、DERIVED（派生表）、UNION 等。这有助于确认查询复杂度。
  - `table`: 访问的具体表名。帮助分析数据来源是否正确。
  - `type`: 表的访问方式，数据读取效率的重要指标，常见的访问类型如下:
    - `ALL`: 全表扫描，访问效率最低。
    - `INDEX`: 利用索引扫描，而非全表扫描。
    - `RANGE`: 范围扫描，索引范围搜索效率良好。
    - `REF`: 基于索引的匹配。
    - `EQ_REF`: 强索引匹配，效率高。
    - `CONST`: 常量值匹配，高效。
  - `possible_keys`: 可能使用的索引，有助于分析是否存在可优化的索引。
  - `key`: 实际使用的索引。确认索引是否正确应用。
  - `key_len`: 索引的长度，表示索引使用的部分字段。例如，对于联合索引，可以看到索引是否全部用上。
  - `ref`: 索引与哪个字段或常量进行了匹配。
  - `Extra`: 包含各种额外的信息，包括 `Using index`（索引覆盖）或 `Using where`（需要计算筛选条件，但无索引覆盖）等。

- 分析具体的优化点：
  - 是否有 全表扫描 (`ALL`)：如有，则需确认索引是否构建合理。
  - 是否正确使用索引：如果 `possible_keys` 有值但 `key` 为空，则索引可能未被利用。
  - 是否存在 `Using filesort` 或 `Using temporary`：
    - `Using filesort`: 表示排序非通过索引完成，需要磁盘文件排序，效率较低。
    - `Using temporary`: 表示创建中间临时表存储数据，通常与分组（`GROUP BY`）或子查询有关。
  - `SELECT` 的字段与索引覆盖：
    - 如果多数字段不在索引中，会导致数据需要从表中回表（即索引无法覆盖整个 `SELECT` 字段集）。
    - 尽量设计索引覆盖查询，以实现更快的查询响应。
  - 返回结果与数据量分析：
    - 数据量：SQL 查询数据量较大时，需关注分页（`LIMIT/OFFSET`）的使用情况，对复杂聚合操作进行优化。
    - 索引匹配度：要确保查询列与索引设计完全一致，防止隐式转换影响性能。
    - 过滤效率：查看 `filtered` 值是否明显不足，可能表示 `WHERE` 条件非高效。
  - 针对当前 SQL 的具体分析重点：
    - 检查索引的应用情况：
      - 根据 SQL 字段传入值和字段类型判断是否发生隐式类型转换导致索引失效
      - `WHERE` 子句中是否包含函数运算，导致索引失效

#### 2. SQL 写法审查与重写规则 (SQL Rewriting Rules)

根据以下规则检查原始 SQL，**仅当能确保语义完全一致时才进行重写**。若无法保证等价，仅指出问题而不改写。记录SQL匹配的规则，输出到最终的优化文档中，命中规则要匹配后才能输出，如无法匹配则输出“无匹配规则”；需逐个匹配所有规则，不能只匹配一两条。

  - **rule01. 投影下推（Projection Pushdown）**  
    仅返回外部查询中实际需要的列，减少不必要的数据传递，仅select * 时命中改规则。  
    例：替换 `SELECT *` 为明确的列选择。
  - **rule02. 选择条件下推（Selection Pushdown）**  
    将 `WHERE` 条件尽量下推到子查询或表的级别，过滤掉不必要的数据。  
    例：将 `WHERE` 条件前移到子查询中，从源表直接过滤无关行。
  - **rule03. 连接条件优化（Join Optimization）**  
    减少连接的数据量或提高连接顺序的效率。  
    例：采用抗数据倾斜的方式对大表连接，或者使用嵌套循环优于笛卡尔积。
  - **rule04. 索引优化（Index Optimization）**  
    确保列上有适合的索引，提高数据访问速度。  
    例：为 `WHERE` 条件的过滤列创建索引，为连接列添加复合索引。
  - **rule05. 分区优化（Partitions Optimization）**  
    利用分区表减少数据扫描范围。  
    例：按日期、区域等维度创建分区，优化大表查询。
  - **rule06. EXISTS 与 IN 优化（Subquery Optimization）**  
    根据场景选用 `EXISTS` 或 `IN`，避免子查询返回大量数据影响效率。  
    例：优先使用 `EXISTS` 处理大表子查询，避免使用 `IN` 时数据量过大导致性能下降。
  - **rule07. 聚合、排序下推（Aggregation and Sorting Pushdown）**  
    在子查询中提前聚合或排序。  
    例：减少 `GROUP BY`、`ORDER BY` 的行数处理。
  - **rule08. 避免函数操作列（Avoid Functions on Columns）**  
    在 `WHERE` 条件中避免对列使用函数，否则会阻止索引发挥作用。  
    例：避免 `WHERE YEAR(column) = 2023`，改用范围查询 `WHERE column BETWEEN '2023-01-01' AND '2023-12-31'`。
  - **rule09. 分组优化（GROUP BY Optimization）**  
    优化分组操作，提前减少无关数据或者替换复杂计算。  
    例：减少参与分组的行数，或者剔除不能索引的冗余列。
  - **rule10. 重复或太复杂的子查询优化（Subquery Deduplication)**  
    对重复出现的子查询进行合并，或分解为更高效的查询。  
    例：在一个查询中多次调用相同的但计算复杂的子查询，可以将其抽取为公用表表达式（CTE）。
  - **rule11. 避免使用 `ORDER BY RAND()`**  
    这种排序方式会对全表的每行调用随机函数，极大影响性能。  
    替代方法：使用伪随机值或者数据库自带方法排序。
  - **rule12. 使用 `LIMIT` 限制数据量**  
    避免对全表操作，适当地限制返回的行数。  
    例：`SELECT * ... LIMIT 100;`
  - **rule13. 避免多表嵌套和太复杂的嵌套查询**  
    尽量简化联表操作逻辑，使用临时表、视图或其他方式避免深度嵌套。
  - **rule14. 加速多表操作的分组统计（Group by Index/Partition Keys）**  
    在多表计算时，优先让分组或聚合操作基于主键或分区键完成。
  - **rule15. 索引列上的运算导致索引失效**  
    在索引列上的运算将导致索引失效，容易造成全表扫描，产生严重的性能问题。所以需要尽量将索引列上的运算转换到常量端进行，譬如下面的SQL。  
    例：select * from tpch.orders where adddate(o_orderdate,  INTERVAL 31 DAY) =date '2019-10-10'  。
    adddate函数将导致o_orderdate上的索引不可用，可以将其转换成下面这个等价的SQL，以便使用索引提升查询效率。
    select * from tpch.orders where o_orderdate = subdate(date '2019-10-10' , INTERVAL 31 DAY);
  - **rule16. 隐式类型转换导致索引失效**  
    当查询条件中的数据类型和数据库表列的数据类型不一致，会发生隐式类型转换，导致索引失效。  
    例：select count(*) from ORDERS where b_id = 123; 
    如果b_id列的数据类型是CHAR(16)，那么b_id上的索引将不会被使用，导致全表扫描。解决方案通常把123强制换换为CHAR类型（提供该重写建议）
     select count(*) ORDERS where ORDERS.b_id = '123';
  - **rule17. IN子查询重写优化**  
    IN子查询是指符合下面形式的子查询，IN子查询可以改写成等价的相关EXISTS子查询或是内连接，从而可以产生一个新的过滤条件，如果该过滤条件上有合适的索引，或是通过PawSQL索引推荐引擎推荐合适的索引，可以获得更好的性能。  
    (expr1, expr2...) [NOT] IN (SELECT expr3, expr4, ...) 
    IN子查询重写为EXISTS
    譬如下面的IN子查询语言是为了获取最近一年内有订单的用户信息，
    select * from customer where c_custkey in (select o_custkey from orders where O_ORDERDATE>=current_date - interval 1 year)
    它可以重写为exists子查询，从而可以产生一个过滤条件（c_custkey = o_custkey）：
    select * from customer where exists (select * from orders where c_custkey = o_custkey and O_ORDERDATE>=current_date - interval 1 year)

    IN子查询重写为内关联
    如果子查询的查询结果是不重复的，则IN子查询可以重写为两个表的关联，从而让数据库优化器可以规划更优的表连接顺序，也可以让PawSQL推荐更好的优化方法。
    譬如下面的SQL， c_custkey是表customer的主键，
    select * from orders where o_custkey in (select c_custkey from customer where c_phone like '139%')

    则上面的查询语句可以重写为
    select orders.* from orders, customer where o_custkey=c_custkey and c_phone like '139%'
    触发条件
      1.IN子查询不是反向条件
      2.如果子查询的结果集是不重复的，可以重写为内关联


#### 3. 索引与元数据综合分析 (Index & Metadata Analysis)

结合 **执行计划** 和 **建表语句 (DDL)** 进行交叉验证：

- **全表扫描确认**: 若出现 `ALL`，检查是否缺少索引或索引失效。
- **索引有效性**: 对比 `possible_keys` 和实际 `key`。
- **隐式转换检测**: 对比 SQL 中的字面量类型与 DDL 中的字段类型（如字符串字段未加引号，数字字段加了引号），这会导致索引失效。
- **重复索引检查**: **关键约束** —— 在建议创建新索引前，必须检查 DDL 中是否已存在相同或包含该索引的联合索引。**若已存在，严禁建议重复创建**。
- **索引覆盖**: 分析 `SELECT` 字段是否能通过现有索引或建议的新索引实现覆盖（避免回表）。
- **特殊列处理**: 若建议创建联合索引，且表中存在 `is_delete` 等区分度极低的列（仅两个值），必须将其放在联合索引的**最后**。
